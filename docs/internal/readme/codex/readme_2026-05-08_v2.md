## See a Skill Run

```text
/prd "A focus-mode feature for our task app"
```

The agent does not guess what a PRD should include. It reads the PM-Skills PRD skill:

```text
skills/deliver-prd/
+-- SKILL.md                 # what the agent should do
+-- references/
    +-- TEMPLATE.md          # the required output shape
    +-- EXAMPLE.md           # the quality benchmark
```

Then it produces a structured product requirements document with problem framing, success metrics, users, scope, user stories, acceptance criteria, dependencies, risks, and open questions.

That same pattern powers every skill in the library:

```text
/hypothesis "One-page checkout will lift conversion"
/opportunity-tree "Reduce onboarding drop-off"
/okr-writer "Q3 OKRs for the growth team"
/meeting-recap "Transcript from product review"
/mermaid-diagrams "Turn this onboarding flow into a sequence diagram"
/slideshow-creator "Create an exec update from this PRD"
```

No prompt engineering. No "please remember to include..." checklist. The standard lives in the repo.

## Install

Most users should choose one of these three paths.

### Claude Code plugin

Best for Claude Code users who want slash commands everywhere.

```text
/plugin marketplace add product-on-purpose/pm-skills
/plugin install pm-skills@pm-skills-marketplace
```

Try it:

```text
/prd "Search v2 for our marketplace"
```

### Skills CLI

Best for the broadest cross-agent install path.

```bash
npx skills add product-on-purpose/pm-skills
```

Use this for agents and IDEs that support the open `skills` CLI / skills.sh ecosystem.

### Git clone

Best for contributors, forkers, local customization, and agents that discover from `AGENTS.md`.

```bash
git clone https://github.com/product-on-purpose/pm-skills.git
cd pm-skills
```

Cursor, Windsurf, GitHub Copilot, and similar workspace agents can discover the catalog from `AGENTS.md`.

## Why It Exists

AI can write product documents, but generic output is uneven. One PRD misses measurable goals. Another buries scope. A retro becomes a vague summary. A hypothesis forgets the validation path.

PM-Skills fixes the repeatable part of the work. It gives the agent a durable artifact standard, so the PM can focus on judgment: the actual problem, tradeoffs, evidence, constraints, stakeholders, and decisions.

| PM work that repeats | PM-Skills gives the agent |
|---|---|
| Artifact structure | Templates |
| Quality expectations | Worked examples |
| Process sequencing | Workflows |
| Common PM frameworks | Skill instructions |
| Contribution quality | Builder, validator, iterator |

The library follows the [Agent Skills Specification](https://agentskills.io/specification), so the skills are just markdown. You can read them, fork them, edit them, and use them across tools.

## The Library

40 skills, organized around the Triple Diamond product lifecycle plus foundation and utility capabilities.

| Category | Use it for | Skills |
|---|---|---|
| **Foundation** | Cross-cutting PM work | persona, lean-canvas, okr-writer, meeting-agenda, meeting-brief, meeting-recap, meeting-synthesize, stakeholder-update |
| **Discover** | Understand users, market, and stakeholders | competitive-analysis, interview-synthesis, stakeholder-summary |
| **Define** | Frame the problem and opportunity | problem-statement, hypothesis, opportunity-tree, jtbd-canvas |
| **Develop** | Explore solutions and decisions | solution-brief, spike-summary, adr, design-rationale |

| **Measure** | Instrument and evaluate | experiment-design, instrumentation-spec, dashboard-requirements, experiment-results, okr-grader |
| **Iterate** | Learn and adapt | retrospective, lessons-log, refinement-notes, pivot-decision |
| **Utility** | Build and transform artifacts | pm-skill-builder, pm-skill-validate, pm-skill-iterate, mermaid-diagrams, slideshow-creator, update-pm-skills |

### Skill families

Some capabilities are intentionally built as families:

| Family | Skills | Why it matters |
|---|---|---|
| OKR Skills | `okr-writer`, `okr-grader` | Write OKRs at cycle start, score them at cycle close |
| Meeting Skills | `meeting-agenda`, `meeting-brief`, `meeting-recap`, `meeting-synthesize`, `stakeholder-update` | Prepare, run, recap, synthesize, and communicate meetings |
| Skill Lifecycle | `pm-skill-builder`, `pm-skill-validate`, `pm-skill-iterate` | Create, check, and improve the library itself |

## Workflows

Skills are useful one at a time. Workflows are for bigger PM jobs where one artifact feeds the next.

| Workflow | Sequence | Command |
|---|---|---|
| Feature Kickoff | Problem -> hypothesis -> PRD -> stories | `/workflow-feature-kickoff` |
| Customer Discovery | Interviews -> JTBD -> opportunities -> problem | `/workflow-customer-discovery` |
| Sprint Planning | Refinement -> stories -> criteria -> edge cases | `/workflow-sprint-planning` |
| Product Strategy | Competitive context -> stakeholders -> opportunities -> solution | `/workflow-product-strategy` |
| Post-Launch Learning | Instrumentation -> dashboard -> results -> lessons | `/workflow-post-launch-learning` |
| Stakeholder Alignment | Stakeholders -> problem -> solution -> launch readiness | `/workflow-stakeholder-alignment` |
| Technical Discovery | Spike -> ADR -> design rationale | `/workflow-technical-discovery` |
| Lean Startup | Hypothesis -> experiment -> result -> pivot/persevere | `_workflows/lean-startup.md` |
| Triple Diamond | Full lifecycle across six product phases | `_workflows/triple-diamond.md` |

Example:

```text
/workflow-feature-kickoff "Admin approval flow for bulk user import"
```

Guide: [Using Workflows](docs/guides/using-workflows.md).  
Reference: [docs/workflows/](docs/workflows/).

## Contributing

PM-Skills is intentionally curated. New skills should solve a repeated PM job, have a clear output contract, and avoid duplicating an existing skill.

### For a small fix

1. Fork the repo.
2. Create a branch.
3. Make the edit.
4. Run the relevant validator.
5. Open a PR.

```bash
bash scripts/lint-skills-frontmatter.sh
bash scripts/validate-commands.sh
```

### For a new skill

Start with the lifecycle tools:

```text
/pm-skill-builder "A skill that creates a market sizing brief"
/pm-skill-validate "Validate the draft market sizing skill"
/pm-skill-iterate "Apply these validation findings"
```

Then read:

- [CONTRIBUTING.md](CONTRIBUTING.md)
- [Authoring PM-Skills](docs/guides/creating-pm-skills.md)
- [PM-Skill Lifecycle](docs/guides/pm-skill-lifecycle.md)

## FAQ

<details>
<summary><strong>Is this only for Claude Code?</strong></summary>

No. Claude Code has the richest slash-command experience, but the skills are portable markdown. They work through the skills CLI, AGENTS.md discovery, ZIP upload, MCP, or manual copy/paste depending on the platform.

</details>

<details>
<summary><strong>Why not keep every platform setup in the README?</strong></summary>

Because the README should help people decide quickly. Full platform instructions live in [docs/getting-started/platforms.md](docs/getting-started/platforms.md), where they can be updated without turning the front door into a manual.

</details>

<details>
<summary><strong>Can I use this commercially?</strong></summary>

Yes. PM-Skills uses the [Apache 2.0 license](LICENSE).

</details>

<details>
<summary><strong>Can I change the templates?</strong></summary>

Yes. Fork the repo and edit the skill's `TEMPLATE.md`, `EXAMPLE.md`, or `SKILL.md`. That is the point of keeping the library in plain markdown.

</details>

<details>
<summary><strong>Where should I start if I am new?</strong></summary>

Start with one artifact:

```text
/prd "Feature you are considering"
```

Then try one workflow:

```text
/workflow-feature-kickoff "Feature you are considering"
```

Compare the output with the examples in `library/skill-output-samples/`.

</details>

<details>
<summary><strong>What is the difference between a skill and a workflow?</strong></summary>

A skill creates one artifact. A workflow sequences multiple skills so the output of one artifact can inform the next.

</details>
