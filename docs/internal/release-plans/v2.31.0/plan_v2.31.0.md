## Why this, why now

The audit named one systemic root cause behind every P0 and most P1s: **hand-synced derived surfaces**. The repo has already proven the cure. `AGENTS.md` (generated by `scripts/gen-skill-manifest.mjs`, CI-gated behind `--check`) has never drifted, while its hand-maintained siblings (QUICKSTART counts, README catalog tables, the sub-agent compatibility matrix, plugin manifest descriptions, the GitHub About text, the skill-builder inventory) all drifted despite green CI. v2.30.0 (trust repair) fixes the current **instances**. This release removes the **hand-write path**, which is the only move that ends the class instead of hunting instances forever. That is the audit's innovation bet X-8 (zero-drift releases as a named system), and it is scored first because it protects every other claim the project makes.

Two honest counterweights are folded in: (1) a generated front door is only trustworthy if the generator is itself tested and CI-gated on both OS legs, so every new generator ships with a `.test.mjs` and a `--check` tripwire (the same posture as `gen-skill-manifest.mjs`); (2) release-please shifts a load-bearing dependency onto squash-PR-title discipline and onto a token that can trigger downstream CI, both of which are called out explicitly in Risks below and gated by a shadow-then-cutover rollout (ZD-4), not a hard switch.

## Dependencies and sequencing rationale

The phase order in the Execution phases table is not arbitrary; it reflects three hard dependencies and one soft one:

1. **WS-Z2 (generators) precedes WS-Z1 (release-please).** release-please updates the README version badge + version row via `x-release-please-version` annotations, and the generator owns the surrounding catalog/badge blocks. Building the generator and its markers first means the release-please annotations are added to lines the generator already leaves stable, so the two never fight over the same region. Standing up release-please against a still-hand-edited README would just automate a moving target.
2. **WS-Z3 (notes dedup) builds on the WS-Z2 generator.** The recent-releases mirror, the changelog mirror, and the releases-index row are additional emit targets of `gen-derived-surfaces.mjs`; they cannot land before the generator exists.
3. **WS-Z4 (port) depends on the v2.30.0 fixture tree.** The port's acceptance test is behavioral equivalence against the v2.30.0 WS-T9 fixtures. If v2.30.0 has not shipped that fixture tree, WS-Z4 cannot prove the port is safe and must wait (Phase 0 confirms the tree exists).
4. **WS-Z1 cutover (ZD-4) is the last gate, softly.** The shadow phase can run early (Phase 3), but authoritative cutover waits until the generators (WS-Z2/Z3) have removed the hand-edits that a shadow Release PR would otherwise still show as diffs, so the shadow PR is a clean signal.

WS-Z5 (evals) and WS-Z6 (trust) are independent of the automation spine and can proceed in parallel with any phase; they are sequenced late only to keep the critical path (the generator + automation) unobstructed.

## Risks

- **The default `GITHUB_TOKEN` does not trigger other workflows.** PRs opened by release-please with the default token do not fire the `validation.yml` checks, so the Release PR would show **no CI** - exactly the surface where CI matters most. Mitigation: ZD-5 (a fine-grained PAT or a GitHub App token). This is verified release-please behavior, not a guess.
- **Squash-title discipline becomes load-bearing.** release-please derives the version bump and the changelog skeleton from squash-merge PR titles. A non-conventional title (or a `feat:` that should have been `fix:`) mis-bumps the release. Mitigation: ZD-3 (a PR-title lint action) plus the shadow phase catching a mis-bump before authority cutover.
- **First-run history scan.** On first run release-please scans history back to the last tag it recognizes; without a bound it may scan the whole repo. Mitigation: set `bootstrap-sha` (or seed `.release-please-manifest.json` with the current `2.31.0`-predecessor version) so the first run starts from a known point.
- **Protected-branch permissions.** With R-06 branch protection landing in v2.30.0, the action needs `contents: write` + `pull-requests: write` (and `issues: write`) and the token/App must be allowed to push the release branch and open PRs against a protected `main`. Mitigation: grant the workflow permissions explicitly and allow the chosen token in branch-protection settings.
- **Generator lock-in risk.** Making generation the only write path means a generator bug silently ships wrong counts. Mitigation: every generator has a `.test.mjs` and a `--check` tripwire on both OS legs; the grep count-sweep (surface I) stays as a belt-and-suspenders check.
- **Scope size.** Six workstreams is a large minor even before catch-up. Mitigation: the catch-up block (C-14/C-15/C-19/C-20) is expected to be a no-op (owned by v2.30.0) and otherwise splits to v2.31.1; the staged block never blocks the tag.
